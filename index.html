Le problème avec SSH entrant

Ansible repose sur SSH : le Control Node doit joindre chaque hôte sur le port 22. Ce modèle bloque sur des infrastructures modernes.

🔒

NAT / Firewall

Les hôtes en réseau privé n'ont pas de port 22 accessible depuis l'extérieur. Ouvrir ce port représente une surface d'attaque inacceptable.

🏭

DMZ / Cloud privé

Les politiques de sécurité interdisent les connexions entrantes dans les DMZ. Les flux de contrôle doivent partir de l'intérieur.

📡

Edge computing

Des centaines de sites distants avec connexions sortantes uniquement, IPs dynamiques, sans VPN centralisé.

⚖️

Salt Minion ? Non.

Salt résout la connexion mais abandonne la compatibilité Ansible. Impossible de réutiliser vos rôles, collections et inventaires existants.

Ansible Control Node ──SSH──▶ Hôte cible ← BLOQUÉ si hôte derrière NAT/firewall

Le modèle inversé AnsibleRelay

AnsibleRelay inverse le flux de contrôle. L'agent sur l'hôte cible initie une connexion WebSocket persistante vers le serveur relay. Ansible n'a besoin d'atteindre que le relay server — jamais les hôtes directement.

Connection Plugin

Remplace SSH dans Ansible. Vos playbooks existants fonctionnent sans modification — changez juste connection: relay dans ansible.cfg.

Inventory Plugin

Inventaire dynamique automatique. Tous les agents connectés apparaissent dans Ansible avec leurs facts et variables.

NATS JetStream

Bus de messages persistant entre nodes relay. Haute disponibilité native : N nodes relay, routing transparent via cluster NATS.

Sécurité by design

AnsibleRelay est conçu autour d'un modèle Zero Trust. Chaque décision d'architecture a une justification sécurité explicite, documentée dans les specs techniques.

RSA-4096 par agent

Chaque agent génère sa paire de clefs RSA-4096 au premier démarrage. La clef privée ne quitte jamais l'hôte.

Enrôlement contrôlé

La clef publique de l'agent doit être pré-autorisée en base de données avant tout enrôlement. Zéro TOFU (Trust On First Use).

JWT chiffré RSAES-OAEP

Le token JWT retourné à l'agent est chiffré avec sa clef publique RSA. Illisible sans la clef privée, même en cas d'interception.

Blacklist JTI

Révocation immédiate par JTI (JWT ID unique). La connexion WebSocket active est fermée avec le code 4001 — l'agent ne reconnecte pas.

Dual-key JWT

Rotation des secrets JWT sans interruption de service. Grace period configurable : les anciens tokens restent valides pendant la migration.

TLS obligatoire

WSS (WebSocket over TLS) et HTTPS sur toutes les connexions. Terminaison TLS via Caddy. Aucun plain HTTP accepté.

become_pass masqué

Le mot de passe d'élévation sudo/su est transmis via stdin chiffré et masqué dans tous les logs — agent, serveur, plugins Ansible.

AES-256-GCM

Les secrets serveur (RSA keypair, JWT secrets) sont chiffrés en base de données via AES-256-GCM dérivé de RSA_MASTER_KEY.

Fonctionnalités implémentées

AnsibleRelay est développé par phases incrémentales, chacune validée par des tests automatisés et un audit sécurité avant déploiement.

v0.1 — Agent Python MVP
  • Enrollment RSA-4096 : génération clef, POST /api/register, JWT OAEP
  • Connexion WSS persistante avec reconnexion backoff exponentiel (1s → 60s)
  • Exécution de commandes via subprocess (buffer stdout 5 MB, timeout, kill)
  • Transfert de fichiers : put_file et fetch_file (base64, limite 500 KB)
  • Élévation de privilèges : become via stdin, become_pass masqué dans logs
  • Registre async persisté JSON : reprise après redémarrage agent
  • Déploiement systemd : Restart=on-failure, User dédié, NoNewPrivileges
v0.2 — Serveur Python (FastAPI + NATS)
  • API REST FastAPI : /api/register, /api/exec, /api/upload, /api/fetch, /api/inventory
  • NATS JetStream : streams RELAY_TASKS + RELAY_RESULTS, routing inter-nodes HA
  • Auth JWT : rôles agent/plugin/admin, blacklist JTI, révocation WS code 4001
  • SQLite : tables agents, authorized_keys, blacklist avec schéma versionné
  • Admin : POST /api/admin/authorize, POST /api/admin/revoke
  • Inventaire dynamique : format JSON Ansible standard, filtre only_connected
v0.3 — Plugins Ansible
  • Connection plugin : remplace SSH, exec_command / put_file / fetch_file natifs
  • become support : transmission become_pass au serveur relay
  • Inventory plugin : GET /api/inventory → hostvars + groupes Ansible
  • Configuration via ansible.cfg ou variables d'hôte (ansible_connection: relay)
  • Mapping HTTP → AnsibleConnectionError : 503 UNREACHABLE, 504 timeout
v1.0 — Réécriture Go (server + agent + inventory)
  • Serveur Go : binaire 4.65 MiB, 0 restart en production, goroutines
  • Agent Go : 94 tests PASS, 3 agents qualif connectés en continu
  • relay-inventory : binaire Go standalone, 19 tests, format Ansible natif
  • SQLite via modernc (CGO-free), Gorilla WebSocket, JWT Go natif
  • Dockerfile multi-stage golang:alpine → alpine : image minimale
v1.1 — CLI Management
  • CLI cobra intégrée dans le binaire relay-server (zéro binaire supplémentaire)
  • minions : list / get / suspend / resume / revoke / authorize / vars (get/set/delete)
  • security : keys status / rotate --grace Nh · tokens list · blacklist list
  • inventory list --only-connected · server status / stats
  • --format table|json|yaml sur toutes les commandes
  • Dual-key JWT : rotation sans interruption, grace period configurable
  • Message WS rekey : agents migrent vers le nouveau secret sans déconnexion
  • AES-256-GCM : chiffrement RSA keypair serveur et JWT secrets en DB

Démarrage rapide

AnsibleRelay se déploie en quelques minutes avec Docker Compose.

1

Démarrer le serveur relay

git clone https://github.com/CCoupel/Ansible-Relay.git
cd Ansible-Relay/GO

export JWT_SECRET_KEY="$(openssl rand -hex 32)"
export ADMIN_TOKEN="$(openssl rand -hex 16)"
export RSA_MASTER_KEY="$(openssl rand -hex 32)"

docker compose up -d
2

Autoriser un agent (pre-authorize)

# La clef publique est générée par l'agent au premier boot
docker exec relay-api relay-server minions authorize my-host \
  --pubkey "$(cat /path/to/my-host_pubkey.pem)"
3

Installer l'agent sur l'hôte cible

# Sur l'hôte cible (Linux, connexion sortante uniquement)
RELAY_SERVER_URL=wss://relay.example.com \
  ./relay-agent --hostname my-host

# Ou via systemd (production)
systemctl enable --now relay-agent
4

Configurer Ansible

# ansible.cfg
[defaults]
inventory  = relay_inventory.py
connection = relay

[relay_connection]
server_url = https://relay.example.com
token      = ${ANSIBLE_RELAY_TOKEN}
5

Lancer un playbook & gérer les agents

# Lancer un playbook Ansible (aucune modification requise)
ansible-playbook site.yml

# Gérer les agents depuis le container
docker exec relay-api relay-server minions list --format table
docker exec relay-api relay-server server status --format json

# Rotation des clefs JWT (zero downtime, grace 24h)
docker exec relay-api relay-server security keys rotate --grace 24h
